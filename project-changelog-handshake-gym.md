## Execution Mode Policy (applies repo-wide, both squads — set 2026-07-10)
- **Autonomous, PR-only.** Work proceeds without stopping to ask permission for each step, but git changes only ever land as an **opened pull request** — never a direct push to a trunk branch (`main` for the general-app squad, `personal-gym-trainer-freelanz` for this squad).
- **User merges.** PRs are never self-merged going forward; the user reviews and merges themselves. (A prior one-off self-merge of PR #4 predates this policy and was done under explicit one-time authorization — not a standing pattern.)
- **Handshake files: one per squad.** This file tracks the gym squad only; the general app's `project-changelog-handshake.md` tracks that squad only. Don't cross-pollinate state between them.
- Open precision question flagged back to the user: "no push" is being read as *no push to a protected trunk branch* — pushing a topic/feature branch (required to open any PR at all) is assumed still fine. Flagging this reading explicitly in case it's not what was intended.

- 2026-07-15 · orchestrator · BUILD — Team plan: shared-data orgs with owner/admin/staff roles and Stripe per-seat billing (branch `claude/subscription-pricing-features-t8131t`) · Per direct user instruction — "go with Team tier (orgs/seats/roles) — ask me if you need to verify" — continuing the pricing-tier backlog started with Basic/Pro. ## First live-deployment smoke test: two real bugs found, plus LINE billing gap closed

- 2026-07-16 · orchestrator · FIX + BUILD — first-ever live smoke test against a real Vercel deployment (branch `claude/subscription-pricing-features-t8131t`), walking through Neon connection, schema migration, and Stripe test-mode setup interactively with the user. Two real, previously-invisible bugs surfaced immediately, plus one gap the user asked to close on the spot.

  **Bug 1 — Stripe endpoints crashed on Vercel's Edge Runtime.** `api/billing-checkout.js`, `api/billing-portal.js`, and `api/stripe-webhook.js` all import `lib/stripe.js`, which imports the `stripe` npm package — its Node SDK relies on Node core modules Edge Runtime doesn't support. Importing it there crashes at module-load time, before the handler function ever runs, producing a bare platform `500` with **zero outgoing requests logged** — not this codebase's own try/catch's `502` JSON body. Live-tested: clicking "Upgrade to Pro" hit exactly this. Every other `api/*.js` endpoint stays safely on `edge` because none of them import `lib/stripe.js` — only `lib/db.js` (the HTTP-based Neon driver, edge-safe) and Web Crypto (edge-safe). **Fix**: switched those 3 files' `config.runtime` from `'edge'` to `'nodejs'` — Vercel's own documented fix for using the full Stripe SDK. No handler logic changed; all three already use the Web-standard `Request`/`Response` handler shape, which Vercel's Node.js runtime supports identically, so `stripe-webhook.js`'s raw-body signature verification (`request.text()`) is unaffected. **Why no prior test caught this**: `test-stripe-webhook.mjs` runs in plain local Node.js, never Vercel's actual Edge Runtime — this whole class of bug (a runtime/bundler incompatibility) is only observable against a real deployment, which no pass before this one had access to.

  **Bug 2 — `enableCloudBackup()` had no guard for LINE-authenticated accounts, and would 400.** Live-tested: logging in via "Continue with LINE" and clicking Settings ▸ "Enable cloud backup" failed with `400 Missing password hash fields`. Root cause: LINE accounts are `salt: null, hash: null, iters: null` by design (no password at all — `app/app.js`'s `handleLineLoginRedirect()`), but `enableCloudBackup()` unconditionally sent those null fields to `api/auth-register.js`, which requires them. The bug had a sibling that was already correctly guarded: `maybeShowCloudBackupModal()` (the auto-popup) already skipped LINE accounts entirely with a comment explaining exactly why an unconditional "Enable" button would be "guaranteed to fail" — but that guard only covered the popup, not the manual Settings button, which is what the user actually clicked.

With no server account, there's nothing for billing to attach to, and no fix to bug 2 alone could close that — a LINE user had **no path to a paid plan whatsoever**. Asked the user whether to fix this now or backlog it; they chose fix now.

  **The fix, built this pass**: rather than force a second OAuth round trip every time a LINE user wants to enable cloud backup (bad UX, and unnecessary — `api/line-login-callback.js` already does the one real verification against LINE), that same callback now also mints a signed, long-lived (365-day) proof of the identity it just verified — `signLineIdentity()`/`verifyLineIdentity()`, new in `lib/lineLogin.js`, same HMAC-signed-stateless-token shape as this file's existing `state` param and every other signed token in this codebase, reusing `LINE_LOGIN_STATE_SECRET` rather than minting a new env var (same "an existing secret already covers this risk" call this session already made for team invite tokens). A new endpoint, `api/auth-register-line.js`, verifies that token and creates-or-looks-up a `users` row keyed by `line_sub` (a column that already existed in the schema but nothing wrote to until now) — unlike `api/auth-register.js`, it's safe to treat "row already exists" as a normal re-login rather than a 409, since the signed token already proves identity as strongly as a password would; no separate login step is needed. `enableCloudBackup()` now branches: password accounts keep the exact existing flow, LINE accounts call the new `SidekickBackend.registerLine()`, and a LINE account with no stored token yet (the pre-fix edge case) gets a clear "log out and back in with LINE" toast instead of a silent failure. `maybeShowCloudBackupModal()`'s guard was loosened to match — it now shows the popup for any LINE account that DOES have a usable token, not none of them.
